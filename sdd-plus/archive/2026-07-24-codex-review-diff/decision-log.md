# Decision Log

## Change

codex-review-diff

## Decisions

| Date | Decision | Reason | Alternatives Considered |
| --- | --- | --- | --- |
| 2026-07-24 | `--diff` sends the changed files' **current content**, not diff hunks | The field report's evidence: full-file reviews caught a privilege-model bypass (`service_role` retaining direct DML) precisely because the reviewer could see the whole object and ask "what else can write this table?" — hunks alone could not surface that. Deleted paths are listed separately so removals are still visible | Send unified diff hunks (rejected: cheaper but strictly weaker on the class of finding that justified the feature); send both (rejected for v1: roughly doubles tokens for a marginal gain) |
| 2026-07-24 | Auto-discovered secret paths are **skipped + disclosed**; explicitly named ones are **refused** | A stray `.env` in the diff must not sink the whole review — but an operator who *names* a secret path is asking for something forbidden and should be told no. Different intent, different response; neither ever sends the file | Refuse in both cases (rejected: one stray file kills the highest-frequency flow); skip in both (rejected: silently ignoring an explicit request is worse than refusing it) |
| 2026-07-24 | Added a **content-level** secret scan on top of the name guard | Codex's review of this very change: *"a normal-looking file such as `app.py` that contains an embedded token or private key passes the guard"*. The name guard structurally cannot see file contents, and auto-discovery makes that likely rather than hypothetical | Name guard only (rejected: auto-discovery raises the exposure it was already blind to); full entropy/secret-scanner (rejected for v1: false positives would block real reviews — high-confidence patterns only, and the limit is stated) |
| 2026-07-24 | Auto-discovered paths must resolve **inside the repo** | Codex flagged that a changed symlink is followed with no containment check, so `--diff` could read and transmit out-of-tree files. Explicit paths are the operator's own choice; auto-discovered ones are not | Follow all symlinks (rejected: an out-of-repo read/exfil vector); refuse symlinks entirely (rejected: in-repo symlinks are legitimate) |
| 2026-07-24 | A git failure is an **error**, never an empty change set | An invalid `--base` silently reading as "no reviewable changed files" tells the operator their branch is clean when the command actually failed — a misleading result in a tool whose whole thesis is not misleading people | Treat empty output as "no changes" (rejected: that is the bug) |
| 2026-07-24 | Boundary marker escalates until absent from the content | Fixed ``` fences can be closed by the reviewed file itself, letting content escape the data region into the instruction region. The marker is now chosen so the content cannot contain it | Fixed fence (rejected: closable); escape backticks in content (rejected: mangles the code being reviewed) |
| 2026-07-24 | The skip lists ride on **every** return, including failures | The verifier falsified R27 ("No skip SHALL be silent") with a repro: a run that declined to send a `.env` and then hit `too_large` reported only the size error. An early failure is still an outcome the Owner acts on, and "we quietly didn't send your secret" is exactly the thing that must never be quiet | Disclose only on success (rejected: that *was* the bug); a second "audit" call to fetch skips (rejected: a disclosure you have to ask for is not a disclosure) |
| 2026-07-24 | The boundary marker is computed over **paths** as well as content | The fence only escaped scrutiny because I checked the obvious half. A filename containing the marker closes its own fence and the rest of the prompt lands in the instruction region — the same attack, through the half I did not fence. Deleted paths were worse: joined into the preamble undelimited | Sanitize paths instead (rejected: mangles the identifier the reviewer needs to cite); omit paths from the prompt (rejected: findings without file attribution are near-useless) |
| 2026-07-24 | Containment **fails closed** when the repo root is unknown | It fails open in `guard_outbound` nowhere else in this codebase; an unverifiable containment check that proceeds anyway is a check in name only. Auto-discovery only ever runs inside a repo in practice, so the cost of refusing is near zero and the cost of being wrong is exfiltration | Skip only the unverifiable paths (rejected: without a root, *every* path is unverifiable); proceed with a warning (rejected: warnings do not stop bytes leaving the machine) |
| 2026-07-24 | `--base` is validated before it reaches a git argv | The verifier turned `--base` into `--output=<path>` and git **wrote a file**. Argument injection, not a bad-ref error. Operator-invoked and therefore low severity, but a tool whose thesis is "don't mislead people" cannot ship a flag that does something other than what it says | Add `--` and rely on position (rejected: `--output` is consumed before `--`); shell-quote (rejected: there is no shell — the argv is the vector) |
| 2026-07-24 | Review timeout floors at 600s rather than scaling from 240s | Measured, not reasoned: the first dogfood died at 240s *after* completing the review, and a size-scaled retry still gave only 245s for a 32KB payload. Deep review is long-running by nature — payload size is the smaller term | Keep the shared 240s default (rejected: throws away completed work); no cap (rejected: a hung delegate would hang the operator) |
| 2026-07-24 | Secret pattern widened to `sk-[A-Za-z0-9_-]{20,}` | Codex's round-2 review, rated high: `[A-Za-z0-9]` stops at the first hyphen, so the pattern matched the *legacy* key format and missed `sk-proj-…` and `sk-ant-…` — i.e. it caught the rare case and let the common one through. A guard that only covers obsolete formats is worse than none, because it reads as coverage | Keep the tighter class (rejected: misses current keys); add each vendor prefix separately (rejected: a new vendor format silently reopens the hole) |
| 2026-07-24 | `--diff-filter=ACMRTU`, adding type changes and unmerged files | Codex: a tracked file replaced by a **symlink** is a type change (`T`), which `ACMR` excluded — so the single case repo-containment exists to catch was filtered out *before* the guard could see it, and could surface as "no changes". Two guards, and the gap sat in the seam between them | Keep `ACMR` (rejected: the exclusion is exactly the security-relevant case); no filter at all (rejected: deletions must stay a separate list) |
| 2026-07-24 | Ambiguous flag combinations are refused, not silently narrowed | `--diff` with paths, or `--base` without `--diff`, quietly discarded scope the operator explicitly asked for. Silent scope loss is the precise failure this tool exists to prevent; being wrong loudly is the product | Ignore the extra flag (rejected: that is the bug); define combined semantics (rejected: invented meaning nobody asked for) |
| 2026-07-24 | R27 wins over R26 for **deleted secret paths**: the name is withheld from the reviewer, disclosed to the Owner | The verifier found R26 ("deleted paths SHALL be reported and named to the reviewer") and R27 ("never send a secret-bearing path") in direct contradiction, with the code obeying only R26 — a deleted `.env` was named to Codex. When two requirements collide, the one protecting the machine wins; the Owner loses nothing because `deleted` still carries the full list | Filter from the Owner's view too (rejected: the Owner is who needs to know); leave it (rejected: a spec violation you have noticed and shipped anyway is a lie in the spec) |
| 2026-07-24 | Prompt paths are repo-relative, not absolute | The R26 subdirectory fix made `changed_files` return absolute paths, which quietly started shipping `C:\Users\<name>\…` — the OS username — off-machine on every `--diff`. A fix for one leak opened a smaller one, in a packet whose thesis is "auto-discovery never widens what leaves the machine" | Keep absolute (rejected: needless outbound identity); strip only the username (rejected: fragile pattern-matching where a clean relpath exists) |
| 2026-07-24 | A verifier is given a **frozen** tree | I edited `review.py`, the tests, and the normative spec while the verifier was reading them; it nearly reported findings against replaced code, and recovered only by pinning md5s itself. A verdict against a moving target describes nothing, and the failure is silent — the report still *looks* authoritative | Keep working during review (rejected: the report becomes unfalsifiable); re-run after every edit (rejected: unbounded cost for a self-inflicted problem) |
| 2026-07-24 | Two reviewers with different vantages, not one run twice | Empirical, this packet: the `verifier` subagent found **spec violations** (skips silent on failure, path-borne fence escape, subdirectory false-clean) while Codex found **implementation gaps** (dead secret regex, `ACMR` filter seam, argparse stderr leak, stat/open TOCTOU). Near-zero overlap. Neither pass alone would have shipped this safely, which is the concrete case for cross-model review as a rung in `/drydock:verify` | One reviewer, more rounds (rejected: the round-1 verifier passed over the regex four times — more rounds of one vantage does not become a second vantage) |
| 2026-07-24 | Dogfooded the feature on its own implementation before archiving | `review.py --diff` on this very change returned 7 findings, all confirmed real on audit. Using the feature to review the feature is the cheapest possible proof that it works — and it found the security gaps above | Ship after unit tests alone (rejected: would have shipped the symlink and content-secret holes) |

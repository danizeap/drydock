hook-deny-and-powershell

## Decisions

| Date | Decision | Reason | Alternatives Considered |
| --- | --- | --- | --- |
| 2026-07-09 | Migrate git_safety/protect_secrets to the JSON permissionDecision deny (exit 0), not keep exit 2 | Exit 2 is fundamentally incompatible with the `python3 X || python X` wrapper: `||` reads the non-zero exit as launch failure and re-runs on drained stdin, failing open. JSON deny + exit 0 is the protocol packet_guard already uses and is `||`-immune | Change hooks.json to not use `||` (rejected: the `||` is needed for the Windows python3-Store-stub case; a shell that distinguishes "not found" from exit 2 is fragile/non-portable); wrapper script that inspects exit codes (rejected: extra process, platform-fragile) |
| 2026-07-09 | Accept `tool_name == "PowerShell"` natively in the plugin guards, not the reference's relabel-to-Bash | The reference relabels because it is a project-level hook that cannot edit the plugin; the plugin CAN treat PowerShell as a first-class shell tool, which is cleaner and honest (git_safety's tokenizer and the shared secret-path checks already handle PS command strings) | Relabel PS→Bash inside the plugin (rejected: a lie in the payload; the tool_name should stay truthful for the ledger and any future PS-specific handling) |
| 2026-07-09 | Port `_ps_write_targets` verbatim from the tested reference; add `command_write_targets(command, tool_name)` unioning POSIX + PS | The reference is built around the exact false-block traps (destination-is-2nd-positional, explicit-path-then-content, attached-colon, unknown-param-skips-value) and ships 34 passing tests; reinventing it risks a wrongful block on the Owner's real work | Write a fresh PS parser (rejected: re-derives solved traps, unproven); regex-only extraction (rejected: the reference already found regex too blunt for the positional/param rules) |
| 2026-07-09 | Union POSIX + PS extractors for PowerShell tools; POSIX-only for Bash | PowerShell shares `>`/`>>` redirection and `cp`/`mv`/`tee` aliases with POSIX AND has native cmdlets — a PS command needs both; a Bash command needs only POSIX (running the PS extractor on Bash is harmless-conservative but unnecessary) | Always union (rejected: rare over-extraction on Bash `copy`; dispatch by tool is precise); dispatch on shell-detection heuristics (rejected: tool_name is authoritative) |
| 2026-07-09 | Fix the liveness probe to detect a JSON deny; keep it single-interpreter; pin the wrapped chain with a separate pytest | After the migration the deny path exits 0, so `||` never fires — a single-interpreter probe is representative again, and the probe's prior overstatement (exit-2-swallowed-by-`||`) no longer exists. The wrapped-chain guarantee belongs in a test, not in per-session runtime cost | Make the probe run the real `python3||python` shell chain every session (rejected: platform-specific, slower, and unnecessary once deny=exit0); leave the probe checking exit 2 (rejected: it would report every migrated guard as degraded — a false alarm) |
| 2026-07-09 | Shared `emit_permission_deny()` helper in `_drydock_common`, used by the two migrated guards; leave packet_guard's working inline deny as-is | Centralizes the protocol + documents the exit-2 prohibition in one place, reducing future drift; not refactoring packet_guard avoids risking a working deny for a cosmetic DRY win | Inline in each guard (rejected: three copies of a security-critical protocol drift); refactor packet_guard too (rejected: unnecessary risk to a working path) |
| 2026-07-09 | Verifier returned NOT VERIFIED; fixed all three findings before archive | The adversarial verifier found a REAL fail-open the reference (and its 34 tests) missed, plus a non-fatal CI step and a git.exe gap — exactly the catastrophic directions a security change must not ship with | Shipping with the verifier's notes open (rejected: a fail-open in the "lethal" guard is the precise failure this packet exists to fix) |
| 2026-07-09 | Invert the PowerShell param default to FAIL-SAFE: unknown `-flag` consumes nothing; only known value-params (`_PS_VALUE_PARAMS`) skip their value | The reference's "unknown param consumes the next token" let a valueless switch (`-Verbose`/`-Debug`) before the path eat the path token and hide the write (verifier repro: `Set-Content -Verbose .env` allowed). Fail-safe means a switch can never hide a path; the residual risk flips to harmless over-extraction of a non-path token | Just add `-verbose`/`-debug` to the switch set (rejected: leaves every other unknown valueless switch fail-open); keep the reference as-is (rejected: proven fail-open) |
| 2026-07-09 | Fix the CI smoke's non-fatal assertions (arg-based, not piped) and add git.exe detection, though both were flagged "pre-existing/out-of-scope" | The packet's own claim is Windows/PowerShell parity: a CI step that can't fail the build and a `git.exe reset --hard` that bypasses git_safety would make that claim false — a faithfulness issue, not a scope question | Defer both to a follow-up (rejected: they directly undercut this packet's stated goal and the "can't be talked out of" promise) |

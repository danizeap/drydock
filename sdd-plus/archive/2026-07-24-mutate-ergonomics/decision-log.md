| 2026-07-24 | Below-resolution fuel is `null` + a reason, and TOKENS become the primary cost signal | Field report #2: a 181k-token task moved the integer weekly gauge by <1%, so the "authoritative" metric read `0` and looked free. That is the absence-of-evidence-as-reassurance failure — the same class as the boolean gauge and the deletion-only diff, now a third time. And the reporter's reframe is decisive: input cost is a near-fixed repo-ingestion floor, not driven by task size, so the gauge can't discriminate per task but tokens can. I inverted their roles when I called the billing figure authoritative | Report fractional gauge precision (rejected: the source gauge is integer percent — there is no finer number, so "precision" would be invented); accumulate a session delta (rejected: still coarse; a per-task question deserves a per-task answer — tokens); leave it (rejected: `0` is a false-free claim in a tool whose whole thesis is not misleading people) |
| 2026-07-24 | Fuel fields renamed to `fuel_used_before/after_percent` | The reporter had to check twice: `fuel_before_percent: 10` are USED percentages while the gauge elsewhere reports `remaining_percent: 92` — same word, inverted meaning, one struct apart | Document the polarity in the note only (rejected: the field name is where the confusion lives); expose remaining instead (rejected: the cost is a used-delta, so used is the natural frame) |
| 2026-07-24 | `--files` caps are IMPORTED from review, not set to matching literals | The 64KB cap sat below a real 90KB file and was asymmetric with `--diff`'s 256KB — a file could be reviewable but not scopable. Since repo ingestion is the dominant cost, scoping is the only lever on it; it must not fail where it's most needed. Importing review's constants makes drift structurally impossible, not just currently-equal | Raise to a matching literal (rejected: two literals drift; this exact asymmetry was born that way); make it configurable (rejected: a knob for a value that should simply track review) |
| 2026-07-24 | A timed-out run keeps its worktree and never clears the gate | The reporter lost a sweep's 100 insertions because `mutate()` deleted the worktree on any non-ok delegation. Partial work is salvageable, not garbage — and the only reason they recovered theirs is that an *external* kill left the tree behind, which is backwards. But a partial is INCOMPLETE, so it must never read as green regardless of what the partial tests say | Delete on timeout (rejected: that is the bug); keep but let it clear if tests pass (rejected: incomplete work passing a subset of tests is exactly a false green) |
| 2026-07-24 | Raised default timeout to 900s + an operator `--timeout` lever, but NO auto-scale | `--diff` scales its timeout on a fixed payload; mutate has none — Codex reads the whole worktree, whose cost we can't size a priori. A raised default covers the common sweep and the lever covers the outlier, honestly, rather than a fabricated size heuristic | Auto-scale on inlined `--files` bytes (rejected: inline size is not the cost driver — the repo read is); keep 600s (rejected: the reporter's real sweep exceeded it); unbounded (rejected: a hung delegate would hang the operator — clamped to 3600) |
| 2026-07-24 | `--gc` "has work" means uncommitted OR committed-unique, and I strengthened the guarantee rather than narrowing the claim | The verifier caught that `git status` sees only uncommitted changes, so a codex worktree that *committed* its work read as empty and got force-deleted — its commits dangling. The finding offered two ways out: tighten the check, or reword the comment to match the weaker behavior. I took the first: a comment that overclaims is the *third* instance this session of "the code guarantees less than the words say," and the right fix is to make the words true, not to lower them. `git branch --contains HEAD` distinguishes a tip that lives only on codex branches (unique work → keep) from one already on a mainline branch (nothing added → remove) | Reword the comment to "uncommitted" only (rejected: technically honest but leaves a real destroy-your-work trap for the day Codex commits); `git log base..HEAD` (rejected: the base isn't tracked at gc time; `--contains` needs no stored base); do nothing since the workflow never commits today (rejected: "can't happen today" is how tomorrow's data loss ships) |
| 2026-07-24 | The work-check fails SAFE — any probe error keeps the worktree | Same N2 lesson: the cost of wrongly keeping an empty worktree is a stale dir the operator removes with `--cleanup`; the cost of wrongly removing one is destroyed work. Asymmetric, so doubt resolves to keep | Fail toward removal on error (rejected: an error must never escalate to deletion); raise (rejected: one bad worktree shouldn't abort the whole sweep) |
| 2026-07-24 | `--gc` removes only EMPTY codex worktrees; no atexit auto-nuke | An external kill orphaned a worktree with no cleanup path. But the N2 lesson binds here: a worktree holding uncommitted work is the exact salvage we just preserved, so GC keeps-and-reports it and only removes genuinely empty orphans. An atexit/SIGTERM handler that removed the worktree would destroy that salvage on every timeout — the opposite of what N2 fixed | atexit/SIGTERM auto-remove (rejected: races directly against the partial-salvage guarantee); remove all codex worktrees (rejected: nukes salvageable work); manual `--cleanup` only (rejected: doesn't solve discovery of orphans the operator forgot) |
